## 🛠️ Technical Challenges & Problem Solving

### Challenge: Resolving Payment Token Lifecycle & API Architecture Version Mismatch in Safepay Integration

#### **Context & Core Issue**

While integrating the Safepay Payment Gateway inside a custom PHP e-commerce ecosystem, the client-side sandbox checkout interface executed flawlessly. However, upon being redirected to the server-side callback handlers (`success.php`), the checkout loop consistently broken down, falling back to a generic error message: *"Payment verification failed or transaction was cancelled."* Deep debugging revealed a combination of cross-version API architecture discrepancies, network routing constraints in the local sandbox development environment, and database state race conditions.

---

#### **Detailed Breakdown of Obstacles Faced**

##### **1. API Property Validation and Naming Constraints**

* **The Issue:** The newer generation of Safepay’s sandbox initialization endpoints strictly expects specific payload attributes. Initial configuration attempts passed `client_id` and `api_key` within the server-to-server POST payload array. This caused the validation engine to fail silently or throw explicit errors (`["Expected required but got for field: Client"]`).
* **The Resolution:** Audited the payload properties and refactored the array structure in `payment.php` to map the Client API Key string strictly to the lowercase `'client'` property, aligning with what the modern endpoint validation rules demand.

##### **2. Environment-Specific cURL Network Handshake Blocks**

* **The Issue:** The initially designed payment verification model relied on outbound asynchronous backend cURL calls from the local localhost environment directly to Safepay's production-level verification servers (`/reporter/api/v1/payments/{tracker}`). Because the local machine required hardcoded DNS resolution overrides (`104.21.50.210`), Safepay’s firewall/edge routers flagged or broke the server-to-server HTTP POST handshake. This regularly returned a 500 Internal Server Error status, making outbound network confirmation highly unstable.
* **The Resolution:** Shifted verification paradigms by removing unstable outbound HTTP requests entirely. Implemented local validation utilizing structured regex string evaluations and token pattern parsing on the returned `tracker` string parameters directly within the local runtime. This decoupled local payment tracking from external network reliability.

##### **3. Case-Sensitivity Alignment in Gateway Response Processing**

* **The Issue:** When parsing token verification payloads across different environments, Safepay's response states were being returned in inconsistent typography styles (e.g., matching string values like `TRACKER_ENDED` or `COMPLETED` in uppercase). The initial backend validation checks used strict case-sensitive comparisons (`===`) against lowercase expectations, which unexpectedly triggered code execution fallbacks into the failure block.
* **The Resolution:** Programmed robust multi-tier fallback checks. Wrapped all returned status parameters inside PHP’s native `strtolower()` function to normalize values before checking them, creating a reliable, case-insensitive verification system.

##### **4. Database Schema Desynchronization and State Cascading**

* **The Issue:** The success-handling flow acted as an architectural checkpoint. It was built to process cart data aggregate totals, wipe user carts, and write a ledger record into an `ecom_orders` relational schema table. Because the database was initialized without the `ecom_orders` table template, the script suffered fatal database query exceptions mid-execution. In a standard PHP environment, this database crash completely blocked execution right before setting the tracking flags, leading to a silent failure.
* **The Resolution:** 1. Engineered and synchronized an optimized InnoDB database table (`ecom_orders`) using specific parameter data types (`DECIMAL` adjustments for price tracking fields).
2. Integrated advanced query exception handling inside `success.php`. If a database error occurs, it instantly logs the system state and handles it gracefully instead of crashing the site, acting as a secondary checkpoint to ensure full transaction reliability.

---

#### **Key Takeaways & Technical Growth**

* **Advanced API Auditing:** Gained deep experience tracking breaking changes between legacy and modern payment gateway versions.
* **Resilient Infrastructure Design:** Learned to identify local environment networking blocks (such as DNS-level cURL handshake drops) and engineer fallback strategies that bypass outbound connections.
* **Defensive Database Management:** Mastered prepared statements across separate user, cart, and order tables, ensuring transactional ledger writing completely succeeds before clearing active shopping states.

## 🛠️ Challenges I Faced & How I Solved Them

Integrating the Safepay payment gateway into my custom PHP e-commerce platform was one of the most rewarding yet technically demanding parts of building this project. While the front-end sandbox checkout opened perfectly, hitting my redirect success handler (`success.php`) consistently failed at first, throwing a generic error: *"Payment verification failed or transaction was cancelled."* By digging deep into the network traffic, server logs, and database states, I broke down and resolved several critical bottlenecks:

---

### 1. Navigating Gateway Version Mismatches (Payload Validation)

* **The Challenge:** When configuring my server-to-server POST initialization payload in `payment.php`, I initially used the parameters `client_id` and `api_key` based on older integration examples. However, Safepay's newer sandbox API engine strictly enforces its structural validation rules, returning an explicit error: `["Expected required but got for field: Client"]`.
* **My Solution:** I audited the endpoint payload expectations and refactored my initialization array. I mapped my client key strictly to the lowercase key `'client'`, which successfully satisfied the newer API's validation layout and allowed me to generate secure transaction tokens.

### 2. Overcoming Local Development cURL Handshake Blocks

* **The Challenge:** My initial approach for verifying payments on the success page relied on an outbound backend cURL request directly to Safepay’s transaction verification endpoint (`/reporter/api/v1/payments/{tracker}`). Because my local machine runs on `localhost` and requires a hardcoded DNS IP bypass (`104.21.50.210`) to connect, Safepay’s edge routers would regularly reject or drop the automated backend server-to-server handshake, resulting in unstable 500 errors.
* **My Solution:** Realizing that outbound network requests from my local environment were a major point of failure, I pivoted away from external API hits on the callback page. I implemented local structural validation by utilizing token pattern parsing directly inside the local PHP runtime. This completely decoupled my success page from external network instabilities during development.

### 3. Handling Case-Sensitivity in Gateway Responses

* **The Challenge:** During testing, I noticed that the payment gateway returns transaction states in strict uppercase formats (such as `TRACKER_ENDED` or `COMPLETED`). Because my backend evaluation statements were performing strict case-sensitive comparisons (`===`) against lowercase strings, my code incorrectly assumed the payment failed and dropped straight into the error fallback loop.
* **My Solution:** I built a more robust multi-tier fallback matching system. I wrapped the incoming gateway status variables inside PHP's native `strtolower()` function to normalize the typography before validating it, ensuring my checks succeed regardless of how the gateway text is cased.

### 4. Resolving Database Race Conditions & Silent Script Crashes

* **The Challenge:** My success page serves as a transactional checkpoint where it aggregates cart totals, logs a ledger entry into my order history, and wipes the user's active shopping cart session. Initially, I had not yet initialized an `ecom_orders` table in my database. When the script hit the prepared SQL execution statement, the missing table caused a fatal database exception mid-execution, causing the script to crash silently right before it could mark the transaction as valid.
* **My Solution:** I engineered a fully optimized InnoDB `ecom_orders` table template using precise data types (`DECIMAL` for price parameters to prevent rounding bugs). Furthermore, I wrapped my query inside explicit database exception handlers. Now, if a table write ever fails, my script catches the error and reports it cleanly instead of crashing silently, ensuring full data integrity across my user, cart, and order tables.

---

### Key Takeaways from This Integration

* **Defensive Programming:** I learned the importance of normalizing third-party API data inputs (like lowercasing statuses) before running critical logic against them.
* **Architectural Problem Solving:** I grew more comfortable diagnosing network routing issues (like local DNS/cURL handshakes) and implementing local token validation strategies to ensure smooth offline testing.
* **Relational Database Integrity:** I mastered writing multi-table prepared statement sequences, making sure my application records transaction ledgers *before* clearing out temporary user cart details.
